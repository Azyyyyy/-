开发流程¶

1.Gitlab里面创建对应的issue，并打好标签和milestone 2.在uniplan中针对该任务进行对应的创建、排期处理 3.功能开发完毕并通过自测后，gitlab中的issue进行关闭，如果有相关总结性结论，整理到wiki中，添加对应状态url后，关闭gitlab issue 4.开发自测后，提交issue等待验收，通过QA测试后，才验收通过 5.相关Bug提交到Jira中，Bug修改不在uniplan排期之中

提测规范¶

自2018年10月8日起，大家在发起提测报告后，出现了以下问题，严重影响了相关QA同学的测试:

1.开发自己都没有针对自己开发的功能用过一遍就发起提测 2.代码中存在及其明显的P1、P2的BUG，稍微使用就能发现的 3.代码没有打包或者环境根本没有任何准备就发起提测 4.QA测试过程中，随意更新提测环境，导致QA测试终端 5.为了向上汇报或者更新UniPlan的工作项，将开发代码提交作为通过验收的标准 以上行为是对开发人员自己的不负责任，对QA同学的不负责任；将原本 开发应该做到的开发自测排除在整个开发周期的过程，变为了QA发现并提交的Bug

1.UniPlan内，开发认为主线Bug已经修复后，可以提交验收请求；相关验收人在通过开发自测后，才允许通过验收。 2.提测功能开发人员自己使用过，确定没有特别明显的P1、P2的Bug后方可以发起提测 3.发起提测报告后，项目负责人对测试环境负责； 4.非P1、P2类Bug不得在本次QA测试过程中发起提测； 5.如果要重新部署，那么本次提测算不通过，QA予以驳回； 6.任何由开发私自处理测试环境导致QA不可测试的情况，统一按照P1的低级Bug提交Jira 7.自提测报告提交起，测试环境必须可用；任何提测报告发起后，测试环境在提测完成前不可用的均按照P0处理 8.QA有权打回任何不符合以上流程的提测报告 项目负责人对提测报告负责，也要约束相关研发人员提测后随意变更测试环境，并做好监督

P1（Priority=1）属于阻碍性Bug，需要第一时间处理。因此Bug被提交后2小时（当日）之内需要被“解决”。 P2优先级的在8小时之内被“处理”。 P3优先级的在24小时之内被“处理”。 P4优先级的可以根据项目情况，经过技术经理权衡后判断是否修复。

尽量分布式提测，最大以功能模块提交；分阶段提测后，再整体提交功能。 季度、年度绩效都会参考具体规范的执行情况和QA同学的反馈 咱们没有必要自欺欺人，需要延期的任务及时与我沟通即可，没有必要为了让我认为你没有延期而提前点击已完成按钮 不要因为我发了这个邮件就不敢提测了，邮件目的是为了为开发同学自己负责，并减少不属于QA同学的工作量。

系统版本规范¶

主要参考：https://semver.org/

1.采用的版本号为: MAJOR.MINOR.PATCH（主版本号.次版本号.修订号） 2.主版本号只有在发生不兼容、系统架构等变更时修改 3.次版本号针对排期内的主要功能、同一主版本向下兼容的功能性新增 4.修订号，做了向下兼容的问题修复时 5.打包时，例如单测环境，要求打包为：主版本号.次版本号.修订号-BetaN，N表示针对主版本号.次版本号.修订号的第几轮测试 6.准备上线TestPass时，要求打包版本为:主版本号.次版本号.修订号-RCN,N表示第几轮TestPass 7.正式发布: 主版本号.次版本号.修订号 8.每次测试修复的BUG，都放在beta(rc)N+1的版本进行修复

周报¶

周报目前要写在两个位置

gitlab 周报 周报系统iwork 因为其他团队需要对周报进行处理，依赖各位的周报： 1.gitlab周报，需要在每周六中午12点之前完成 2.iwork的周报，需要在每周日中午12点之前完成 尽量详细描述本周工作，避免流水帐
